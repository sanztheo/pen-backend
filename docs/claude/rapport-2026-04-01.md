## DEEP-DIVE : CONCURRENCE & FIABILITÉ

### Méthodologie
Scan exhaustif du codebase pour : race conditions read-then-write, verrous distribués manquants sur crons, JSON merges non-atomiques, patterns d'idempotence, et double-fetch dans les request lifecycles.

### 1. Race Conditions — JSON Merge non-atomique

**#104 — MOYENNE — `controllers/user/personalizationController.ts:125-157`**

Pattern classique read-then-write sur le champ JSON `user.settings` :
```typescript
const existing = await prisma.user.findUnique({ where: { id }, select: { settings: true } });
const currentSettings = (existing?.settings as Record<string, unknown> | null) || {};
const merged = { ...currentPersona, ...personalizationData };
const newSettings = { ...currentSettings, personalization: merged };
await prisma.user.update({ where: { id }, data: { settings: newSettings } });
```

**Risque** : Deux updates personnalisation concurrents (ex: onboarding steps envoyés en parallèle) — le second écrase les changements du premier. Non transactionnel.

**Risque pratique** : Faible (opération user-scoped, usage single-tab typique), mais pattern incorrect qui devrait être corrigé avec une transaction ou un `Prisma.DbNull` merge.

### 2. Verrous distribués — Crons sans Redis NX

Les crons suivants manquent de lock Redis NX, contrairement aux crons critiques (`monthlyReset`, `dailyLimitsReset`, et tous les crons Beta) qui en ont :

| Cron | Fréquence | Risque multi-instance | Mitigation existante |
|------|-----------|----------------------|---------------------|
| **alertsCron** (#105) | Toutes les 5 min | Alertes dupliquées | Cooldown NX dans AlertsService (partiel) |
| **retentionCron** (#106) | Hebdomadaire (dimanche 00:00) | Double compute | `upsert` idempotent |
| **RAG cleanup** (#107) | Planifié | Double compute + double delete | DELETE idempotent |

**Analyse** : En mono-instance, aucun risque. En multi-instance (Railway scale-out), ces crons s'exécutent N fois. Les opérations sont idempotentes (upsert, delete), donc pas de corruption, mais gaspillage de ressources et alertes potentiellement dupliquées.

**Recommandation** : Aligner avec le pattern existant — ajouter `redis.set(lockKey, "1", "EX", TTL, "NX")` comme les autres crons le font déjà.

### 3. Fire-and-Forget Counters

**#108 — BASSE — `controllers/project.ts:116-130`**

```typescript
Promise.all([
  prisma.userLimits.update({ where: { userId }, data: { projectsUsed: { increment: 1 } } }),
  prisma.workspace.update({ where: { id }, data: { lastActivityAt: new Date() } }),
]).catch((error) => {
  logger.error("⚠️ [ASYNC] Erreur updates non-bloquants:", error);
});
```

Le `projectsUsed` increment est fire-and-forget avec un `.catch()` qui ne fait que logger. Si l'update échoue (ex: Neon cold start timeout), le compteur sera définitivement désynchronisé.

**Comparaison** : Le delete (L427-445) utilise correctement `$transaction` avec le decrement. Asymétrie entre création et suppression.

### 4. Yjs Persistence — Flush Race Condition

**#109 — BASSE — `lib/y-prisma.ts:86-107`**

```typescript
async flushDocument(docName: string): Promise<void> {
  const ydoc = await this.getYDoc(docName);       // ← Lit state + all updates
  const newDocState = Y.encodeStateAsUpdate(ydoc); // ← Encode
  // ... findUnique ...
  await prisma.$transaction([
    prisma.yjsDocument.update({ data: { data: newDocState } }), // ← Write merged
    prisma.yjsUpdate.deleteMany({ where: { documentId } }),     // ← Delete ALL updates
  ]);
}
```

**Risque** : Les `storeUpdate()` concurrents entre `getYDoc()` et le `$transaction` créent des updates qui seront supprimées par `deleteMany` sans être intégrées dans le state encodé.

**Mitigation naturelle** : Le protocole Yjs est basé sur CRDT — les clients gardent l'état en mémoire et re-synchronisent automatiquement. La perte est temporaire et auto-guérissante.

### 6. Idempotence — Billing Endpoints

**#110 — MOYENNE — `routes/billing.ts:212-257`** — POST /cancel sans garde idempotente

```typescript
const subscription = await prisma.userSubscription.findUnique({
  where: { userId },
  select: { paddleSubscriptionId: true },
});
// ❌ Pas de check : subscription.cancelAtPeriodEnd === true
await paddle.subscriptions.cancel(subscription.paddleSubscriptionId, { effectiveFrom: "next_billing_period" });
```

**Risque** : Double-click cancel → 1er appel réussit, 2e appel → erreur Paddle (déjà annulé) → user voit une erreur 500 alors que l'annulation a bien fonctionné. Fix simple : vérifier `cancelAtPeriodEnd`/`status` avant d'appeler Paddle.

**#111/#112 — BASSE — `routes/billing.ts` checkout-session & upgrade** — Pas de dedup

Les POST /checkout-session et /upgrade créent des sessions Paddle multiples si double-click. Pas de corruption (Paddle gère), mais gaspillage de ressources. Fix : dedup par `userId + interval` avec time-window.
