## DEEP-DIVE : RESILIENCE & ERROR HANDLING

### Méthodologie
Scan exhaustif de tous les fichiers `src/` (335 fichiers TypeScript) couvrant 8 catégories :
timeouts, silent catches, retry, heavy processing, graceful shutdown, circuit breaker, error consistency, stream handling.

### 1. TIMEOUTS SUR APPELS EXTERNES

**Tous les `fetch()` ont `AbortSignal.timeout()`** — vérifié sur les 20+ call sites :
- OpenAI API calls : 30s timeout (rag/index.ts, ai.ts, documentSearchService.ts)
- Wikipedia API calls : 15s timeout (wikipediaSearch.ts, functions.ts, webTools.ts, wikipedia.ts)
- Mem0 API calls : timeout configurable `MEM0_TIMEOUT_MS` (mem0Client.ts)
- Paddle API : 5s timeout (index.ts health check)
- Clerk SDK : 10s via `withTimeout` (auth.ts)
- Paddle SDK : 15s via `withTimeout` (billing.ts, paddleWebhooks.ts)
- AI SDK : timeouts configurés (90s fast, 180s thinking, 300s streaming)

**Verdict : COMPLIANT** — Tous les appels externes sont protégés.

### 2. SILENT CATCHES

167 blocs `catch` analysés. Aucun `catch (e) {}` vide trouvé.

Pattern dominant : `catch (error) { logger.error("...", error); }` suivi d'une réponse structurée ou d'un re-throw. Quelques catches légitimes sans log :
- `EmailService.ts:212` — retry queue, incrémente counter
- `conversationService.ts:243` — fallback JSON parse -> UIMessage v5 valide
- `encryption.service.ts:219` — retourne false (tentative de déchiffrement)
- `blocknote.ts:323,659,682` — fallback parsing (JSONL malformé, format auto-détection)

**Seule anomalie :** `blocknote.ts:682` — catch vide sur ServerBlockNoteEditor sans logging → issue #103

**Verdict : ACCEPTABLE** — Les catches sont documentés et intentionnels.

### 3. RETRY & BACKOFF

Infrastructure solide :
- `lib/retry.ts` : exponential backoff avec jitter correct
- `retryPrismaOperation()` : baseDelay 150ms, maxDelay 1500ms
- `retryPrismaTransaction()` : baseDelay 200ms, maxDelay 2000ms

Aucun retry agressif sans backoff trouvé.

**Verdict : COMPLIANT**

### 4. HEAVY PROCESSING IN REQUEST CYCLE

Les opérations AI lourdes utilisent le streaming SSE (pas de blocage). BullMQ est utilisé pour :
- Email queue (rate limiting)
- Potentiellement d'autres jobs async

Le quiz streaming (`quizStreaming.ts`) effectue la génération AI directement dans le request handler, ce qui est correct pour le SSE streaming pattern.

**Verdict : ACCEPTABLE**

### 5. GRACEFUL SHUTDOWN

**`index.ts:796-814`** gère SIGTERM/SIGINT :
- Stoppe monitoring, alerts cron, retention cron
- Stoppe BullMQ workers
- Ferme BullMQ queues

**`lib/prisma.ts:171`** gère séparément la déconnexion Prisma.

**Manquant :**
- `redis.quit()` — la connexion Redis principale n'est pas fermée → issue #102
- `wss.close()` — le serveur WebSocket n'est pas arrêté gracieusement
- `server.close()` — le serveur HTTP continue d'accepter des connexions pendant le shutdown
- Pas de drain des requêtes in-flight

**Verdict : PARTIEL** → issue #102

### 6. CIRCUIT BREAKER

**`lib/circuitBreaker.ts` (119L)** implémente un circuit breaker correct avec 3 états (CLOSED/OPEN/HALF_OPEN), failure threshold 3, reset timeout 30s, probe-in-flight flag.

**Problème : ce module n'est importé par AUCUN autre fichier.** Le `PennoteAgent.ts` utilise `resolveModel` pour le provider selection, mais les services externes (OpenAI direct, Wikipedia, Mem0, Tavily) n'ont aucune protection circuit breaker.

Si OpenAI tombe pendant 5 minutes, chaque requête va timeout à 30s au lieu de fail-fast. À 100 users concurrents = 100 connexions bloquées.

**Verdict : NON-COMPLIANT** → issue #101

### 7. ERROR RESPONSE CONSISTENCY

Pattern dominant : `res.status(500).json({ error: "message" })`. Pas de format standardisé unifié mais acceptable pour la taille actuelle du projet. Le `catch-500 dupliqué` est couvert par l'issue existante #62.

**Verdict : ACCEPTABLE** (couvert par #62)

### 8. STREAM ERROR HANDLING (SSE)

**Agent chat (`routes/agent/chat.ts:257-297`)** :
- `pipeUIMessageStreamToResponse()` gère le streaming
- `consumeStream()` appelé avec `.catch()` pour les erreurs non-fatales (reasoning-delta ordering)
- Géré par le AI SDK v6 — cleanup automatique

**Autocomplete (`controllers/ai/autocomplete.ts:75-80`)** :
- `req.on("close")` correctement implémenté
- `AbortController.abort()` annule l'opération AI
- Pattern exemplaire

**Quiz streaming (`controllers/quizStreaming.ts`)** :
- SSE headers configurés correctement
- `sendSSE()` helper avec `res.flush()`
- **AUCUN `req.on("close")` handler** — si le client ferme le tab, l'opération AI continue jusqu'à completion/timeout
- Gaspillage de crédits AI et de ressources serveur

**Verdict : PARTIELLEMENT COMPLIANT** → issue #100
