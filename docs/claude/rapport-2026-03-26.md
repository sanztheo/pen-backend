## DEEP-DIVE : BASE DE DONNÉES & SCALABILITÉ

Premier audit dédié de ce domaine. Scan exhaustif de 141+ fichiers TypeScript, 2 schemas Prisma, et toutes les queries du codebase.

### 1. Pagination manquante — Le problème #1

**Le module quiz est un champ de mines.** 7 méthodes de `StatsService` et 2 dans `quizService` chargent TOUS les quizzes d'un utilisateur sans aucune limite. Avec `include: { result: true }`, le champ `detailedScoring` (JSON massif) est chargé pour chaque quiz.

**Impact à 10k users** : un power user avec 500+ quizzes → dizaines de Mo de JSON par requête stats.

Détail :

| Fichier | Méthode | Problème |
|---------|---------|----------|
| `statsService.ts:139` | `getAdvancedUserStats` | Tous quizzes de la période sans take |
| `statsService.ts:209` | `getProgressionOverTime` | idem |
| `statsService.ts:238` | `getSubjectBreakdown` | idem |
| `statsService.ts:312` | `getDifficultyAnalysis` | idem |
| `statsService.ts:345` | `getTimeAnalytics` | idem |
| `statsService.ts:409` | `getPageSourcesUsage` | idem |
| `statsService.ts:481` | `getQuestionTypeStats` | idem |
| `quizService.ts:897` | `getQuizHistory` | **Params `limit`/`offset` déclarés mais jamais utilisés dans la query !** |
| `quizService.ts:1543` | `getUserProgressStats` | Tous quizzes complétés, 3 tables jointes |

**Autre pagination manquante critique :**
- `workspace.ts:244` — `getWorkspaces` charge **tous les projets avec toutes les pages** (include tree 3 niveaux) — appelé à chaque navigation sidebar
- `conversations.ts:276` — GET `/messages` charge TOUS les messages d'une conversation
- `conversations.ts:575` — Token counting charge TOUS les messages avec `thinking`, `toolCalls`, `intermediateThinkingBlocks` (colonnes JSON massives)

### 2. Index manquants — Full table scans silencieux

**Page model** — Le plus critique. La query `WHERE workspaceId = X AND isArchived = false` est utilisée 20+ fois dans le codebase (sidebar, projets, pages). Aucun index sur `workspaceId` ni `projectId` sur le model Page. Seul `createdBy` est indexé.

| Model | Index manquant | Queries affectées |
|-------|---------------|-------------------|
| **Page** | `[workspaceId, isArchived]` | 20+ (sidebar, projets, workspace) |
| **Page** | `[projectId, isArchived]` | 10+ (pages par projet) |
| **Project** | `[workspaceId, isArchived]` | 5+ (listing projets) |
| **QuizSequence** | `[userId]` | `getQuizHistory` |
| **AIConversation** | `[userId, workspaceId, isActive]` | Conversations actives par workspace |
| **ActivityLog** | `[userId, createdAt]` | Queries admin temporelles |

### 3. N+1 Queries

| Fichier | Problème | Impact |
|---------|----------|--------|
| `page.ts:835` | deletePage: boucle `for pid of allIds` → `removeUserPage()` séquentiel | 50 sous-pages = 50 appels RAG |
| `page.ts:931` | cleanupArchived: même boucle séquentielle | Idem |
| `rag/cleanup.ts:134` | processBatch: 2 queries/source (deleteMany chunks + delete source) | Devrait être batch DELETE ... IN |
| `rag/userPages.ts:452` | Chunk INSERT: `$executeRaw` individuel par chunk en boucle | 100 chunks = 100 round-trips au lieu de 1 batch INSERT |

### 4. Intégrité transactionnelle

| Fichier | Problème | Risque |
|---------|----------|--------|
| **`page.ts:196`** | `pagesUsed` incrémenté fire-and-forget hors transaction après `page.create` | Counter désynchronisé si `Promise.all` échoue silencieusement |
| **`simplifiedContent.ts:211`** | `projectsUsed` incrémenté fire-and-forget `void (async () => ...)` | Idem |
| **`simplifiedContent.ts:376`** | `deletePage` ne décrémente JAMAIS `pagesUsed` | Counter ne fait qu'augmenter — `controllers/page.ts` le fait correctement |
| `page.ts:640` | Slug generation check-then-act sans transaction | 2 requêtes parallèles → même slug (impact faible — slug informatif) |

### 5. Include trees lourds

| Fichier | Query | Profondeur |
|---------|-------|------------|
| **`workspace.ts:244`** | getWorkspaces: workspace → projects → pages + pages racine + members | **3 niveaux** — toutes les pages de tous les projets |
| **`workspace.ts:342`** | getWorkspaceById: même structure | **3 niveaux** |
| `conversations.ts:575` | Token counting: conversation → ALL messages (avec JSON massifs) | 2 niveaux mais volume énorme |
| `statsService.ts` | Tous les quizzes avec `include: { result: true }` | 2 niveaux — JSON `detailedScoring` |

### 6. Points positifs

- **Connection pool** : bien configuré (100 connections prod, 30 dev, timeout 30s, keepalive 5min)
- **Raw queries** : toutes utilisent les tagged template literals Prisma (paramétrisées, 0 risque injection SQL)
- **Aucun `$executeRawUnsafe`** dans le codebase
- **Graceful shutdown** : `$disconnect()` appelé correctement

## TOP 5 ACTIONS PRIORITAIRES

1. **Ajouter les index Page** `[workspaceId, isArchived]` et `[projectId, isArchived]` — gain de perf immédiat, risque quasi nul, impacte 20+ queries (#71)

2. **Paginer StatsService + quizService.getQuizHistory** — le `limit`/`offset` est déjà dans la signature, il faut juste l'utiliser dans la query. Remplacer `include: { result: true }` par `select` ciblé (#68, #69, #70, #89)

3. **Refactorer getWorkspaces** — charger les pages par projet à la demande (lazy) ou ajouter `take` sur les pages imbriquées (#72, #82)

4. **Mettre les counter increments dans des transactions** — `createPage`/`createProject` doivent être atomiques avec l'incrément, et `simplifiedContent.deletePage` doit décrémenter (#80, #81)

5. **Batch les INSERT RAG chunks** — INSERT multi-values au lieu de `$executeRaw` en boucle (#78)
