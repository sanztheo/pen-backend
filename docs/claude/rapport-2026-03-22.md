## SCALABILITÉ & BASE DE DONNÉES (Objectif 10k)

- **Index manquant sur Quiz pour les requêtes agent** — `quizTools.ts:36-44` et `:61-76` filtrent sur `(userId, isCompleted, completedAt)` mais le modèle Quiz n'a que `@@index([userId])` (`schema.prisma:334`). À 10k utilisateurs avec des centaines de quiz chacun, ces requêtes feront un index scan sur `userId` puis un filtre séquentiel sur `isCompleted` + `completedAt`. **Action : ajouter `@@index([userId, isCompleted, completedAt])`.**

- **3 requêtes DB séquentielles dans `getQuizStats`** — `quizTools.ts:36`, `:47`, `:61` exécutent 3 queries Prisma en séquence (aggregate, aggregate, findMany). Elles pourraient être parallélisées avec `Promise.all()` puisqu'elles sont indépendantes. Latence inutilement doublée pour chaque appel tool.

- **`getUserConversationDetail` charge 500 messages sans pagination** — `adminUserDetailController.ts:83-97`. Chaque message inclut `toolCalls` et `pageCreationData` (colonnes Json potentiellement volumineuses). Pas de curseur ni de paramètre `before`/`after`. À 10k users avec des conversations longues, payload de plusieurs Mo possibles.

- **`getUserQuizDetail` retourne `questions` et `userAnswers` Json sans projection** — `adminUserDetailController.ts:200-201`. Ces champs Json peuvent contenir des dizaines d'objets complexes. Aucun garde-fou sur la taille du payload.

- **`checkInactiveUsers` — scan sans index composé** — `BetaCronService.ts:42-56`. Filtre sur `betaStatus = 'active' AND (lastHeartbeatAt < threshold OR ...)`. Seul `@@index([betaStatus])` existe, pas de composé `(betaStatus, lastHeartbeatAt)`. Full scan de tous les users actifs.

- **`processWaitlist` — N transactions sérialisables séquentielles** — `BetaCronService.ts:126-147`. Chaque promotion de waitlist = 1 transaction sérialisable avec retry + backoff. Si 50 places disponibles = 50 transactions séquentielles. Correct pour la cohérence, mais lent et consommateur de connexions pool.

- **`workspaceTools.ts:104` — `findMany` sans `take`** — `listWorkspacePages` ne limite pas le nombre de pages retournées. Un workspace avec 500+ pages retournera tout en mémoire puis au LLM.

- RAS sur `getRecentQuizResults` — correctement borné par `take: limit` avec max 20 via Zod.

## SÉCURITÉ & OWASP

- **`sortBy` non validé dans `getBetaUsers`** — `adminBetaController.ts:67`. Casté en `string` et passé directement au service. Devrait être un `z.enum()` des colonnes autorisées. Risque : Prisma ne fait pas de SQL injection mais un champ invalide provoque une erreur non gérée.

- **`userId` et `action` query params non validés dans `getModerationLogs`** — `adminUserController.ts:55-56`. Cast brut `as string`, pas de vérification de format. Le `userId` devrait être validé comme les params (pattern existant : `validateUserId`).

- **`toggleUserStatus` n'utilise pas Zod** — `adminUserController.ts:79-89`. Vérification manuelle `typeof isActive !== "boolean"` alors que tous les autres write endpoints du même fichier utilisent des schemas Zod. Incohérence qui fragilise le contrat.

- **Impersonation sans rate limit dédié** — `admin.ts:81`. `POST /impersonate/:userId` partage le même `adminRateLimit` (150/15min) que les endpoints de lecture. **Un rate limit dédié et strict (ex: 10/heure) est nécessaire pour cette action sensible.**

- **Export admin sans rate limit par utilisateur** — `adminExportController.ts:29-96`. Le middleware `accountExportRateLimit` existe dans `rateLimiting.ts` mais n'est pas appliqué à cette route. Un admin peut déclencher des dizaines d'exports qui scannent toute la base.

- Quiz tools : pas de risque IDOR — `ctx.userId` vient de la session authentifiée et filtre toutes les requêtes. RAS.
- Auth middleware sur toutes les routes admin : `authenticateToken` + `requireAdmin`. RAS.

## CONCURRENCE & FIABILITÉ

- **`checkInactiveUsers` sans verrou distribué** — `BetaCronService.ts:35-97`. Contrairement à `cleanupExpiredAccounts` (ligne 197) et `sendPositionUpdates` (ligne 366) qui prennent un lock Redis `NX`, `checkInactiveUsers` n'a aucun guard. **En scaling horizontal (2+ instances), deux crons concurrents écriront des `reactivationDeadline` différents et dupliquent les logs d'audit.**

- **`resetWeeklyCounters` sans verrou** — Même pattern. L'`updateMany` est idempotent donc le double-run est sans danger fonctionnel, mais génère du bruit en logs et des écritures DB inutiles.

- **Position update : merge JSON non-atomique** — `BetaCronService.ts:464-478`. Le pattern `{ ...user.existingMeta, lastNotifiedPosition }` spread un snapshot capturé au moment du read. Si `metadata` est modifié entre le read et le write, les modifications sont silencieusement écrasées. Race condition sur les mises à jour concurrentes de metadata utilisateur.

- **`requireAdmin` — double fetch DB par requête** — `requireAdmin.ts:24-27` fait un `findUnique` puis `checkAdminStatus` en fait un autre (`adminDashboardController.ts:29-32`). Le résultat `isAdmin` n'est pas attaché à `req.user`, forçant un re-fetch. Sur un dashboard admin avec 10 appels parallèles = 20 queries redondantes.

- Quiz tools : gestion d'erreur correcte avec try/catch + log + réponse structurée. RAS.

## QUALITÉ & TESTS

- **Pas de tests pour `quizTools.ts`** — 300 lignes de logique métier (calcul de streak, trend, agrégation par sujet) sans aucun test unitaire ou d'intégration. Les fonctions `calculateStreak`, `calculateTrend`, `extractSubjects` sont des candidats idéaux pour des tests unitaires (logique pure, pas de side effects).

- **`cacheBlockNoteContent` sans timeout** — `adminUserDetailController.ts:285-286`. Appel Redis sans `AbortSignal.timeout()`. Si Redis hang, l'endpoint admin bloque indéfiniment. Les conventions du projet exigent un timeout sur tout appel externe.

- **`JSON.parse(JSON.stringify({ filters }))` dans export** — `adminExportController.ts:79`. Méthode fragile pour créer un objet sérialisable. Crash silencieux sur référence circulaire. Un commentaire ou une approche explicite serait préférable.

- **Refactoring admin : bonne qualité** — Le split de `adminController.ts` (1535 lignes) en 5 fichiers spécialisés (<300 lignes chacun) suit les conventions du projet. Séparation claire des responsabilités.

- `adminUserDetailController.test.ts` — 792 lignes de tests ajoutés. Couverture correcte du controller détail.
