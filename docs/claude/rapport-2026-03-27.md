## DEEP-DIVE : SÉCURITÉ & OWASP

### Méthodologie
Scan systématique de toutes les routes (`src/routes/**/*.ts`) et services pour :
- Couverture `authenticateToken`
- IDOR (accès cross-tenant)
- Validation des inputs (Zod vs raw)
- Rate limiting par route
- SQL injection (Prisma.raw)
- Exposition de données sensibles

### Résultats

#### 1. Auth Middleware — Couverture complète
Toutes les routes appliquent `authenticateToken` soit :
- Via `router.use(authenticateToken)` au niveau du router (user, quiz, workspace, agent, etc.)
- Via middleware par route (auth, billing, content, upload)
- Via montage dans `index.ts` (assistant, agent)

**Exception légitime :** `GET /api/quiz/stream/:sessionId` — SSE ne supporte pas les headers Auth, l'auth est gérée via `?token=` dans le contrôleur (vérifié).

**Exception problématique :** `GET /api/upload/config` (l.210) — **Pas d'auth**. Expose config serveur (types autorisés, taille max).

#### 2. **IDOR CRITIQUE — `PUT /api/content/projects/:id`**
- **Fichier :** `routes/content.ts:238`
- **Problème :** `prisma.project.update({ where: { id } })` — pas de filtre `createdBy: userId`
- **Impact :** N'importe quel utilisateur authentifié peut modifier n'importe quel projet (titre, nom, metadata) s'il connaît l'UUID
- **Comparaison :** La route `PATCH /projects/:id/pin` (l.348) fait correctement `findFirst({ where: { id, createdBy: userId } })` avant l'update
- **Fix recommandé :** Ajouter `prisma.project.updateMany({ where: { id, createdBy: userId }, data: {...} })` ou check `findFirst` avant

#### 3. Rate Limiting — Gaps significatifs
**Routes AVEC rate limiting dédié :**
- `/api/auth` → `authRateLimit`
- `/api/ai`, `/api/agent` → `aiRateLimit` + `aiBurstRateLimit`
- `/api/assistant` → `assistantRateLimit` + `aiBurstRateLimit`
- `/api/quiz` → `quizRateLimit`
- `/api/admin` → `aiRateLimit`
- `/api/feedback` → `feedbackRateLimit` (interne)
- `/api/agents` → `generatePromptRateLimit` (seulement generate-prompt)

**Routes SANS rate limiting dédié (seul `globalRateLimit` s'applique) :**
- `/api/billing` — **checkout, cancel, upgrade, portal-url** — routes financières sensibles
- `/api/upload` — **upload fichiers vers Cloudinary** — abus = coût direct
- `/api/content` — CRUD projets/pages
- `/api/conversations` — CRUD conversations + generate-title (appel LLM)
- `/api/ai-credits` — deduct, refund
- `/api/quiz-limits` — deduct, refund
- `/api/limits`, `/api/sync-limits`, `/api/workspaces`, `/api/projects`, `/api/pages`

#### 4. Input Validation — Zod limité à 4 fichiers
**Fichiers avec Zod :** `content.ts`, `agents.ts`, `feedback.ts`, `ai.ts`
**Fichiers sans Zod sur inputs :** `billing.ts` (priceId/interval bruts), `aiCredits.ts` (validation manuelle OK), `quizLimits.ts`, `limits.ts`, `upload.ts`, `conversations.ts`

- `billing.ts:84` — `priceId` et `interval` acceptés depuis `req.body` sans validation Zod
- `aiCredits.ts:26,126` — Validation manuelle du montant (OK, mais Zod serait plus cohérent)

#### 5. SQL Injection — Risque théorique via Prisma.raw
- `rag/index.ts:825` et `wikipediaTools.ts:365` utilisent `Prisma.raw(\`'${embeddingStr}'::vector\`)`
- `embeddingStr` = `queryEmbedding.join(",")` — valeurs float de l'API OpenAI
- **Risque :** faible (source fiable) mais pas de validation que les valeurs sont bien des nombres
- **Fix recommandé :** Ajouter `if (!queryEmbedding.every(v => typeof v === 'number' && isFinite(v))) throw`

#### 6. Données sensibles — Pas d'exposition majeure
- Les error stacks ne sont jamais envoyés dans les réponses HTTP (juste loggés)
- Zod validation errors exposent les champs invalides (acceptable)
- `billing.ts:106-108` logge `priceId` et `email` — acceptable pour debug billing
