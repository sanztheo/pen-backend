## DEEP-DIVE : CONCURRENCE & FIABILITÉ

Scan exhaustif du codebase pour race conditions, verrous distribués manquants, idempotence, état partagé mutable, et fiabilité des queues.

### **🔴 CRITIQUE — Race conditions sur opérations financières**

#### #29 — `refundCredits` read-then-write (aiCreditsService.ts:214-248)

```typescript
const userLimits = await prisma.userLimits.findUnique({ where: { userId } });
const newAiCreditsUsed = Math.max(0, userLimits.aiCreditsUsed - amount);
await prisma.userLimits.update({ where: { userId }, data: { aiCreditsUsed: newAiCreditsUsed } });
```

**Contraste flagrant** : `deductCredits` dans le même fichier utilise un UPSERT SQL atomique avec `CASE WHEN`, mais `refundCredits` retombe sur le pattern read-then-write classique. Deux remboursements concurrents (scénario réaliste : agent AI qui fail avec retry) peuvent perdre ou gonfler les crédits.

**Fix :** `$executeRaw` avec `SET ai_credits_used = GREATEST(0, ai_credits_used - ${amount})`

#### #30 — `deductAdvancedQuiz` read-then-write (quizLimitsService.ts:690-710)

```typescript
const userLimits = await prisma.userLimits.findUnique({ where: { userId } });
await prisma.userLimits.update({
  data: { advancedQuizzesUsed: userLimits.advancedQuizzesUsed + 1 }
});
```

Même pattern. Bypass trivial : 10 requêtes parallèles depuis le frontend, toutes lisent `advancedQuizzesUsed = 9`, toutes écrivent `10`. L'utilisateur consomme 10 quiz pour le prix d'1. `deductCustomQuiz` dans le même fichier utilise un UPSERT atomique — incohérence.

**Fix :** `$executeRaw` avec `SET advanced_quizzes_used = advanced_quizzes_used + 1 WHERE advanced_quizzes_used < advanced_quizzes_limit`

#### #31 — `processWaitlist` sans verrou Redis (BetaCronService.ts:163-231)

Toutes les autres méthodes du BetaCronService (`checkInactiveUsers`, `resetWeeklyCounters`, `cleanupExpiredAccounts`, `sendPositionUpdates`) acquièrent un lock Redis NX. `processWaitlist` est la seule exception. En multi-instance, deux backends exécutent le cron simultanément → double envoi d'emails de promotion (non revertible même si la DB rejette les doublons via transaction Serializable).

**Fix :** Ajouter le même pattern `redis.set(lockKey, "1", "EX", TTL, "NX")` que les autres méthodes.

### **🟠 HAUTE — Fenêtres de race et patterns non-atomiques**

#### #32 — `lastAICallTime` TOCTOU (futuraRss.service.ts:336-354)

Variable `let` module-level lue puis écrite après un `await delay()`. Deux appels concurrents contournent le rate limiter AI car les deux lisent le même timestamp, attendent, puis écrivent. En Node.js single-threaded le yield est au `await` — c'est suffisant pour intercaler deux callers.

#### #33 — Deep workflows hardcoded Google (workflows.ts:64-76)

`requireGoogle()` throw si `GEMINI_API_KEY` absent. Aucun failover vers OpenAI/DeepSeek. Le chat agent (`PennoteAgent.ts`) a son propre circuit breaker multi-provider, mais les workflows (deep research, deep content) sont isolés sur un seul provider.

#### #34 — Double lecture `canUseAI` + `deductCredits` (requireAICredits.ts:61-77)

`canUseAI` fait un `findUnique` pour vérifier les crédits, puis `deductCredits` refait un read via l'UPSERT. La vérification préalable est **redondante** car `deductCredits` retourne `{ success: false, limitReached: true }` si la limite est atteinte. Elle ajoute une fenêtre de race et une requête DB inutile par requête AI.

#### #35 — `canCreatePresetSequence` check/create (quizLimitsService.ts:274-363)

Le check fait un `findFirst`, puis `startPresetSequence` fait une transaction Serializable. Sous charge modérée, deux requêtes passent le check avant que la première transaction ne commit. La transaction Serializable mitige partiellement, mais n'est pas infaillible si les deux transactions ne chevauchent pas.

#### #36 — `resetMonthlyCredits` legacy (aiCreditsService.ts:372-401)

Read-then-write non protégé. Non appelé par le cron principal (qui utilise du SQL batch), mais existe comme méthode publique invocable. Idempotent (reset à 0 deux fois = même résultat), mais architecturalement dangereux si réutilisé ailleurs.

### **🟡 MOYENNE — Fiabilité et design gaps**

#### #37 — `getWeeklyArticle` findFirst+create sans transaction (futuraRss.service.ts:638)

Cron + admin refresh concurrent → deux articles identiques créés. Pas de contrainte unique DB sur la semaine.

#### #38 — Circuit breaker HALF_OPEN (circuitBreaker.ts:43)

Quand le circuit passe de OPEN à HALF_OPEN, tous les callers async concurrents sont admis au lieu d'un seul probe. Contrevient au design pattern circuit breaker.

#### #39 — Turnstile fail-open (waitlistController.ts:15-23)

`TURNSTILE_SECRET` capturé au module-load. Si absent, `verifyTurnstile` retourne `true` → CAPTCHA désactivé silencieusement. Devrait throw au startup.

#### #40 — Email queue en mémoire (EmailService.ts:107-156)

`emailQueue: QueuedEmail[]` process-local. Crash = emails perdus. Multi-instance = pas de partage. Le Redis pending store ne couvre que les 429 Resend, pas le flux normal.

#### #41 — Prompt sans max-length sur `/workflow` (routes/agent.ts:571)

Seul `!prompt` est vérifié. Un prompt de 100MB passe, est slicé à 3000 chars dans le workflow mais consommé entièrement côté quota (`prompt.length / 4`). Vecteur de quota bypass.

#### #42 — Paddle webhook idempotence TOCTOU (paddleWebhooks.ts:168-176)

`findUnique` + traitement + `create` au lieu de `create`-first avec catch `P2002`. Deux livraisons parallèles du même webhook passent toutes les deux le check.

#### #43 — Metadata spread après TTL lock (BetaCronService.ts:462-469)

Le seeding des positions initiales accumule des updates avec `{ ...meta, lastNotifiedPosition }` lus hors transaction. Si le lock Redis expire (TTL 5min) et un second run démarre, les spreads peuvent écraser des valeurs.
