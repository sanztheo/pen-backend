## SCALABILITÉ & BASE DE DONNÉES (Objectif 10k)

- **Index composite manquant sur `quizzes`** — `getQuizStats` exécute `WHERE userId = ? AND is_completed = true AND completed_at >= ?` mais le seul index est `@@index([userId])`. À 10k utilisateurs avec des centaines de quizzes chacun, Postgres fera un index scan + filter au lieu d'un index-only scan. **Recommandation : ajouter `@@index([userId, isCompleted, completedAt])` sur le modèle Quiz.**

- **3 requêtes DB séquentielles dans `getQuizStats`** — `prisma.quiz.aggregate()`, `prisma.quizResult.aggregate()`, et `prisma.quiz.findMany()` sont exécutées en série. Elles pourraient être lancées en parallèle avec `Promise.all()` pour diviser la latence par ~3. À 10k utilisateurs concurrents, cette sérialisation triple le temps de holding de connexion DB.

- **`findMany` avec `take: 50` — OK** — La pagination est correctement bornée. Pas de risque de requête unbounded.

- **Agrégation `scoreStats` via relation imbriquée** — `prisma.quizResult.aggregate({ where: { quiz: { userId, isCompleted, completedAt } } })` force une sous-requête avec jointure. Prisma génère un `WHERE quiz_id IN (SELECT id FROM quizzes WHERE ...)`. Acceptable en volume modéré mais pas optimal. Alternative : requête raw SQL avec `JOIN` si les perfs se dégradent.

- **Traitement en mémoire (`subjectMap`, `calculateStreak`, `calculateTrend`)** — Opère sur max 50 résultats grâce au `take: 50`. Pas de risque mémoire. RAS.

## SÉCURITÉ & OWASP

- **Contrôle d'accès (IDOR) — OK** — Toutes les requêtes filtrent par `userId: ctx.userId` provenant du contexte d'authentification. Un utilisateur ne peut pas accéder aux stats d'un autre.

- **Validation d'input — OK** — Les paramètres sont validés par Zod (`z.enum`, `z.number().min(1).max(20)`). Pas d'injection possible via les inputs.

- **Rate Limiting — Non applicable directement** — Ces tools sont appelés par l'agent AI, pas exposés comme endpoints HTTP. Le rate limiting est géré au niveau du endpoint `/chat` qui invoque l'agent. RAS.

- **Pas de fuite de données sensibles** — Les logs n'exposent que `userId` et des métriques agrégées, pas de PII. RAS.

- **Cast `as string[]` sur `strengths`/`weaknesses` (lignes 119, 123)** — Le champ JSON est casté sans validation Zod. Si la donnée en DB n'est pas un `string[]` (ex: objet, nombre), le code propage silencieusement des données incorrectes plutôt que de crasher. Risque faible car le `Array.isArray()` protège, mais un schéma Zod serait plus robuste.

## CONCURRENCE & FIABILITÉ

- **Pas de mutation — RAS** — Les deux tools sont read-only. Pas de race condition possible. Pas de transaction nécessaire.

- **Gestion d'erreurs — OK** — Chaque tool a un `try/catch` qui log l'erreur et retourne un objet structuré de fallback. L'agent AI peut continuer à fonctionner même si un tool échoue.

- **Pas de timeout sur les requêtes DB** — Si Postgres est sous charge, les 3 requêtes séquentielles de `getQuizStats` pourraient bloquer le stream de l'agent pendant plusieurs secondes. Recommandation : ajouter un `AbortSignal.timeout(5000)` ou un timeout Prisma au niveau de la transaction (mais Prisma ne supporte pas le timeout par requête nativement — à monitorer).

## QUALITÉ & TESTS

- **Pas de tests d'intégration** — Aucun test pour `quizTools.ts`. Les helpers (`calculateStreak`, `calculateTrend`, `extractSubjects`, `getTopFrequent`) sont des fonctions pures facilement testables. Recommandation : ajouter des tests unitaires au minimum pour ces helpers.

- **Fichier à 300 lignes — limite** — Exactement au seuil de la convention projet (max 300 lignes). Si d'autres tools quiz sont ajoutés, séparer les helpers dans un fichier `quizToolsHelpers.ts`.

- **`extractSubjects` — cast `as { name: string }`** — Le type guard `"name" in s` est correct mais le cast pourrait être remplacé par du narrowing plus strict. Risque faible.

- **`calculateStreak` — bug potentiel subtil** — L'itération utilise `uniqueDays.size` comme borne de boucle, mais le `Set` est construit à partir de `dates` qui est pré-filtré. Si aucun quiz n'a été fait aujourd'hui, le streak sera 0 même si l'utilisateur a fait des quizzes les 30 jours précédents. C'est probablement intentionnel (streak = jours consécutifs incluant aujourd'hui), mais cela mérite un commentaire explicatif.
