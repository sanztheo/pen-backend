## RESUME

Migration complète du pipeline quiz de Kimi (kimi-k2.5) vers **Gemini 3.1 Flash Lite** (generation, correction, extraction, clustering, explanations). Ajout de 6 endpoints admin pour le detail utilisateur (conversations, quizzes, pages, AI usage). Nouveau cron job pour emails de progression waitlist. Refactoring correction quiz avec batch explanations Gemini en parallèle.

## SCALABILITE & BASE DE DONNEES (Objectif 10k)

- ~~**`sendPositionUpdates()` — N+1 critique sur `betaWaitlist.update()`**~~ ✅ **CORRIGÉ (PR #86)** — `findMany` paginé par batch de 500 (`WAITLIST_PAGE_SIZE`). Le N+1 sur le seeding initial persiste (one-shot acceptable) mais la pagination élimine le risque mémoire.

- **`getUserConversationDetail` — messages limités à `take: 500`** : OK pour le plafond, mais les messages incluent `toolCalls` et `pageCreationData` (JSON potentiellement lourd). A 10k users, un admin qui navigue dans les conversations peut générer des payloads > 5 MB. **Recommandation :** Paginer les messages (take: 50 + page) au lieu de retourner 500 d'un coup.

- ~~**`getUserAIUsage` — `$queryRaw` avec `GROUP BY DATE`**~~ ✅ **VÉRIFIÉ (PR #86)** — Index composite `@@index([userId, createdAt])` déjà présent sur `openai_usage_log`. Rien à ajouter.

- **`getUserQuizDetail` — `questions` et `userAnswers` en JSON** : Retourne l'intégralité du JSON `questions` et `userAnswers` sans limite. Pour un quiz de 50 questions avec matching + options, le payload peut dépasser 1 MB. RAS pour l'admin (usage faible), mais à surveiller.

- **`correctionGenerator` — tri O(n²) avec `findIndex`** : `sortedCorrections.sort()` utilise `questions.findIndex()` dans le comparateur — O(n²) pour n questions. Acceptable pour les quiz actuels (< 50 questions), mais créer une `Map<id, index>` serait trivial.

- **`questionGenerator` — token budget optimisé** : Le passage de 4096 tokens fixes à un budget adaptatif par type (MCQ: 1500, MATCHING: 3000) est un bon gain de latence et coût. RAS.

- **Quiz pipeline timing logs** : Les `logger.info()` ajoutés dans `quizStreaming.ts` sont utiles pour le monitoring mais ajoutent du bruit en production. **Recommandation :** Utiliser un niveau `debug` ou un flag configurable pour ces métriques de pipeline.

## SECURITE & OWASP

- **Routes admin protégées** : Tous les nouveaux endpoints passent par `authenticateToken` + `requireAdmin` + `adminRateLimit` via le `router.use()` global dans `admin.ts`. RAS.

- **Validation des params** : `userId.length > 255`, pagination plafonnée à 100, `parseInt` avec fallback NaN. RAS — pattern cohérent avec les endpoints existants.

- **`getUserAIUsage` — `$queryRaw` avec paramètre bindé** : Le `userId` est passé via template literal Prisma (`${userId}`), qui utilise le binding paramétré — pas d'injection SQL. RAS.

- **`getUserPageContent` — contrôle d'accès** : Vérifie `page.createdBy === userId` avant de retourner le contenu. Pas de risque IDOR. RAS.

- **`provision-single-user.ts` — mot de passe en clair dans les logs** : Le script affiche `Password: ${password}` dans stdout. Script CLI one-shot, mais si lancé dans un CI avec logs persistés, le mot de passe temporaire sera loggué. **Recommandation :** Masquer après la première ligne ou logger dans un fichier dédié.

- **`send-welcome-test.ts` — mot de passe hardcodé** : `temporaryPassword: "Pennote-test1234!"` en dur. Script de test, mais si commité en production, un attaquant connaît le format des mots de passe temporaires. **Recommandation :** Utiliser un generateur aléatoire comme dans `provision-single-user.ts`.

- **Email `from` changé à `noreply@pennote.fr`** : Bon changement par rapport à `onboarding@resend.dev`. Vérifier que le domaine `pennote.fr` a bien les enregistrements **SPF, DKIM, DMARC** configurés chez Resend pour éviter le classement en spam.

- ~~**Gemini API key fallback**~~ ✅ **CORRIGÉ (PR #86)** — `getGeminiInstance()` lance maintenant une erreur si `GEMINI_API_KEY` est absent au lieu de basculer silencieusement sur OpenAI.
